纸质书？电子书？

发表于 2015-07-07   |   分类于 思考   |  

前天把最后两本书拿去退，才意识到虽然还未毕业，但大四不在学校了（预计），也就可能再也没有机会去图书馆借书了。如此想来，依稀有些伤感，因为我的书单上还有很多书没有读过，很多读过的书也都是囫囵吞枣，实在是遗憾。

现在朝书架上看去，有很多买过的书，都是大学这几年自己买的，随手一列，大概有：

  • 《苏菲的世界》
  • 《1984》
  • 《数学之美》
  • 《程序员修炼之道》
  • 《http权威指南》
  • 《锋利的jQuery》
  • 《python核心编程》

还要其他一些，不一一列了。

惭愧地说，没有一本是从头到尾看完的，大多是随手翻翻，阅过某些章节，还有一些从未看过。如今要考虑的问题是：毕业了，如何处理？

这些大多是我喜爱的书，视之如瑰宝，自然不愿意贱卖掉。那么毕业之时搬运也是一个问题。现在很流行用户体验这个词，从便携这个角度上来说，纸书的用户体验是比不上电子书的。据说有很多人在下定决心在有自己的房子之前，绝对不买纸质书籍，一律看电子书，因为倘若时常需要搬家，那么搬运纸质书绝对不是一个好的用户体验。

那么电子书有没有可能取代纸质书呢？

事实上我看电子书有一段时间了。大多数时候我用kindle在看小说，在那块6寸的屏幕上看过了金庸、古龙、余华等等，kindle毋庸置疑是看文字类图书的好工具。既能提供纸质书的阅读体验，又能随声携带。但是仅限于文字类的图书，如果是专业书籍，就显得不足，几乎没办法看。在电子书领域里，现在多看阅读做的非常好，特别是在排版上，上面很多专业书籍都做无可挑剔，似乎是一个看专业书籍的好选择。我在上面买了几本技术书籍，都很满意。唯一的问题就是：你可能需要一块大屏幕才能看的舒服，比如我在一块5寸的屏幕上看书就非常累，大脑也不愿意持续地看。所以如果你有一个平板电脑，最好是4:3的屏幕，那估计效果会好很多。

电子书的另外一个问题，就是资源少，不是每一本纸质书都有对应的电子书。实际上根据我的估计，两者的比例可能远远小于1:10。也就是说，你想看一本书，不一定有电子版，也就是说还是要买/借 纸质书才能看。

当然网上有很多盗版的PDF文档，大多是有人照着纸质书扫描，然后上传共享的。如果运气好的话，可能扫描的还算清晰，可以一读，但多数情况扫描出来的质量很糟糕，读起来很不方便。而且这里可能涉及盗版的问题。

说了这么多，你问我兹持电子书还是纸质书，那我当然兹持电子书。只是电子书还有很长的路要走。

使用vagrant统一开发环境

发表于 2015-06-28   |   分类于 编程   |  

在开发的时候，通常遇到的一个问题是——为什么在我的机器上是正常的，在你那就不能跑了。一般原因无非就是开发环境不同意，缺少某个库、组件、依赖，所以并不是「一次编写，到处运行」，而是「一次编写，到处调试」。特别是在团队合作开发，或者跨平台开发的时候，经常遇到。

那么有没有可能同意开发环境，让每个人的机器都一样的呢？答案就是vagrant。在vagrant里面，就有这一个环境，大家可以都在这个环境里开发，这样就统一了。我也是刚刚了解到这个概念，斗胆一说。

首先：因为我也是初次尝试（而且目前也没人合作开发：）），这里就不列具体命令了，而且这些东西都很容易在网上搜索到。这里只是谈一下我自己的理解。

0，vagrant依赖于VMware或者virtual box之类的虚拟机软件，说到底它就是在虚拟机里构建了一个操作系统。

1，在vagrant里，有一个box的概念，这个box就是一个操作系统（通常是一个linux），你可以去网上下载box（很容易找到），然后安装整个box。这样你就有了第二个操作系统。

2，然后在vagrant这个系统里，你可以搭建你需要的环境，比如你要开发PHP，那就可以把lamp整套装好，各种库，驱动，依赖装好。（这里也可以用网友配置好的box，可以直接使用）

3，好了之后，就可以利用vagrant里的环境进行开发了。这里有一个同步文件夹的概念，可以把vagrant里的文件映射到本地的某个文件夹，所以你可以在宿主系统中，使用你习惯的编辑器 / ide开发，而运行用的是vagrant里的环境。

4，是不是很酷，是！

5，那你要问，如何统一开发环境呢。答案就是vagrant里面的系统可以打包带走。假如这次使用的是ubuntu，明天又使用windows开发，或者团队里来了新成员，都可以把你的vagrant里的系统打包好，然后到windows/新成员的电脑里，重新部署。这样的开发环境就统一了——都用的是vagrant里的环境。

如何成为一名黑客

发表于 2015-06-16   |   分类于 思考   |  

《如何成为一名黑客》是著名程序员Eric Steven Raymond写的一个小册子，在早上读完了它，在此记录要点，并加以一些我的理解（可能会有偏差）。

需要说明的是，本文只是笔记，记录了一些我觉得值得关注的要点，如果有人看到这里，我建议还是读原文，很容易在网上找到中文原文：）

什么是真正的黑客

一般大众认为黑客是专门搞破坏的人，实际上那些只能称之为骇客（cracker）。我个人这都是拜媒体宣传所赐。真正的黑客是有指有能力对电脑进行操作，并有建设的人。

一言以蔽之：黑客搞建设，骇客搞破坏。

如何能成为一名黑客

  • 对世界充满好奇（世界上充满了迷人的问题）
  • 一个问题不应该被解决两次 （如果你解决了某个不常见的问题，应该把他们分享出来（不违反商业利益的前提下），帮助别人解决同一个问题）
  • 不要做重复性的工作（想办法把无聊的、乏味工作自动化）
  • 不盲从权威，崇尚自由（所以程序员都是讨厌防火长城的）
  • 态度不能替代能力

如何提升能力

  • 学会编程 （不要浮躁，十年学会程序设计？）
  • 语言和框架会阻碍思考（要保持开放的心态，不局限于某个语言和框架）
  • 尝试不同的编程范式（c/c++ python java lisp perl ）
  • 读代码（没什么好解释的）
  • 写代码（没什么好解释的）
  • 使用*Unix系统（这一点我暂存异议，因为对于新手来说，使用Windows的学习成本，比类Unix的学习成本低很多）
  • 使用Web把知识、新的分享出去
  • 学好英语（程序的世界就是英语的世界！！！）

其他项

  • 编写开源软件
  • 给开源软件做共享
  • 流畅用母语写作
  • 阅读科幻小说
  • 学习武术
  • 练习冥想
  • 不要卷入网络争论（骂战）

JavaScript 原型链

发表于 2015-05-22   |   分类于 前端   |  

理解原型对象

创建一个对象，就会根据特定的规则为这个对象创建[[prototype]]属性，这个属性指向函数的原型对象，同时原型对象又有一个constructor属性，指向这个对象。

关于[[prototype]]，是无法直接访问的，但是在各个浏览器的实现中，可以通过__proto__去访问。
例如：

function a(){}
a.__proto__
//function Empty() {}

又可以通过isPrototypeOf()方法来判断原型对象与实例之间是否存在关系

Function.prototype.isPrototypeof(a)
//true

又可以通过Object.getPrototypeOf()来取得某个对象的原型:

Object.getPrototypeOf(a)
//function Empty() {}

还可以通过hasOwnProperty()检测实例的某个属性是否来自实例本身。

function Person(name){
this.name = name;
}
Person.prototype.age = 28;
p1 =new Person("mike");

console.log(p1.hasOwnProperty("name"));
//true 来自实例自身
console.log(p1.hasOwnProperty("age"));
//false 来自原型

但是当hasOwnProperty()为false时，却不能说明属性就来自原型（有可能是属性不存在）。那么可以配合in操作符来判断。

console.log("name" in p1);
//true 来自实例本身
console.log("age" in p1);
//true 来自原型

也就说，不管属性来自实例，还是原型，只要能找到这个属性，in 判断就返回true。

所以，我们可以判断，在属性 in 实例返回true，并且hasOwnProperty()返回false的情况下，这个属性就来自原型。

原型继承

面向对象语言都有三大特性:封装，继承，多态。JavaScript中继承又与其他面向对象语言有些不同，它的继承是通过原型链来实现的。

只需看一段代码，就能大致了解原型继承是什么了：

function father(){
this.father = "father";
}
function son(){
}
son.prototype = new father();
var foo = new son();
console.log(foo.father);
//father

在这里首先定义了两个构造函数father 和son。我们知道，创建一个函数，函数内部会有一个[[prototype]]属性指向他的原型对象，当实例需找某个属性是，如果在自身中找不到，就会沿着[[prototype]]找下去。

这里我们把son的prototype属性，手动的指向father的实例（重写了原型对象），这里的巧妙之处就体现了：当son中寻找某个属性时，会沿着prototype找下去，而这里prototype指向了father！

所以打印foo.father时foo能够找到father中的father这个属性，继承也就这样实现了。

而所谓原型链就是通过对象之间的[[prototype]]的指向，从而实现继承的一个关系（我自己的理解），例如所有的对象对直接或间接的继承了Object(所以所有的对象都有toString hasOwnProperty等方法，都是从Object继承而来了)。

JavaScript几种常见的设计模式

发表于 2015-05-22   |   分类于 前端   |  

工厂模式

function factory(name,age,sex){
var o = new Object();
o.name = name;
o.age = age;
o.sex  = sex;
return o
}

var a = factory("jack",25,"man");
var b = factory("mike",20,"women");

在这里factory函数就是一个工厂，给它传递三个参数，就会返回一个包含有这三个函数的对象，就不用每次手动去生成对象了。
但是工厂模式有一个缺点：不能识别对象的类型。

构造函数模式

function Person(name,age,sex){
this.name = name;
this.age = age;
this.sex = sex;
this.say=function(){
    console.log("hello");
}
}
var p1 = new Person("张三",12,"man");
var p2 = new Person("李四",15,"man");

这里与工厂模式的区别是，直接把age,name ,sex这些付给this对象，那么当我通过 var p1 = new Person() 创建对象的时候，就创建了一个Person类型的对象，并且把参数作为对象的属性赋值给了p1。

那么function Persion(),就作为p1,p2的构造函数，所以p1,p2就有一个constructor属性指向Person

console.log(p1.constructor);
//[Function Person]

与工厂模式相比，就能识别出对象的类型，例如这里就能识别出对象是一个Person类型。

值得注意的是，如果构造函数不是通过new来使用，而是直接调用，那么它与普通函数没有什么区别，this就指向当前环境的对象（如果在对象中，就是某个对象如果在全局中，就指向window）

构造函数的缺点，需要多次创建函数对象。那么可以考虑，能否将每个对象的共同部分抽象出来。那么原型模式就呼之欲出了。

原型模式

要使用原型模式，首先要知道原型是什么意思，具体的可以看我上一篇的总结。这里简单说一下。

当我们创建一个对象时，系统会自动为这个对象创建一个原型对象，并且这个对象的[[prototyype]]属性，指向这个原型对象，与此同时，原型对象里又有[[constructor]]属性指回这个对象。

当对象寻找某个属性是，先在自己的属性里面查找，如果找不到，就向它的原型对象查找。

举个例子：

function Person(name){
this.name = name;
}
Person.prototype.age = 28;
var p1= new Person("mike");
var p2 = new Person("jack");
console.log(p1.age);//28
console.log(p2.age);//28

在这里首先定义了Person函数，然后通过Person.prototype.age = 28 为它的原型对象添加一个属性age，并设置为28。然后在Person的实例p1,p2中访问age属性，实际上并没有，但是它们能通过向上一级，查找原型对象的属性，所以能打印出28。

这里值得注意的是无论是p1,p2还是以后的pn,都共享了age=28这个属性，也就是说如果：

Person.prototype.age=29;

那么p1,p2的age也会变为29。

这与Java之类的语言中的static这个概念很类似，可以把二者联系起来类比。

还需要注意的是，如果单方面修改实例的值，是为实例添加某个属性，而不会影响原型中的值。看代码：

console.log(p1.age);
//28 这里的值来之Person.prototype

Person.prototype.age=29;
//修改原型中的值为29

p1.age=30
//p1并没有age这个属性，这一句实际上是为p1加上age这个属性并赋值30。并不影响原型中的age
console.log(p1.age);
//30
//这里因为p1已经有age这个属性了，不用向原型中查找，所以直接打印30

console.log(p2.age);
//29
//而p2还需要向原型中查找，而原型中的age并改为了29,所以打印29。

介绍到这里，就可以解决了构造对象模式的缺点，把实例之间所共有的属性加入原型对象中，就不用每次创建对象是赋值共有属性了。

其他设计模式

这只是最基础的三种设计模式，还有很多的设计模式，但几乎都是这三种的变种或组合，我自己还没有很深的体会，暂且不表。

JavaScript作用域链与闭包

发表于 2015-05-21   |   分类于 前端   |  

执行环境
首先有一个执行环境（execution context）的概念，每一段代码都有自己的执行环境，这个执行环境对应着一个变量对象,每执行一段代码，就会把这段代码的环境压入环境栈中，执行之后再推出栈。

作用域链
正如上面所说，函数在执行时，会创建一个执行环境,已经对应的作用域链，并且把内部的[[scope]]属性指向这个作用域链，并且把this，arguments，函数的参数来初始化函数的活动对象（变量对象），在作用域链函数的自身的活动对象在第一位，外部的活动对象在第二位，外部的外部在第三位……依次类推。如果要寻找某个变量，如果在自己的活动对象里找不到，就依次向上一级寻找。

并且要特别注意，函数的作用域链，在函数的定义时就已经决定了。

这个说有点抽象，举个例子。
定义一个最简单的函数：

var foo="foo";
function test(){
var foo = "bar";
console.log(foo);
}
test();
// bar

在这个函数里，因为它是在全局中定义，所以在定义这个函数时就决定了它的作用域链是这样的：
作用域链的第一位是函数的活动对象，这个活动对象有this ,arguments ,a 等参数。又因为是在全局作用域中定义的，作用域链的第二位，是全局的变量对象，包含全局作用域中的属性已经a 等等。

在执行的时候，就会把这个作用域链压入到环境栈中。所以，在这个例子中，打印foo，就会优先从函数的活动对象找到foo 。所以是bar。

但是如果是这样:

var foo="foo";
function test(){
// var foo = "bar";
console.log(foo);
}
test();
// foo

就会打印foo.

闭包

闭包是指有权访问另一个函数作用域中的变量的函数。
这么说不好理解，举个例子：

var obj={
test : "Jobs"
}

function fuc(para){
return function(Object){
    return Object[para];
}
}

var a = fuc("test");
console.log(a(obj));
//Jobs

在这里，首先定义一个函数fuc，它返回一个匿名函数。
然后用调用这个函数，并用a接收函数的返回值，也就是说现在a指向这这个返回的函数，通过a(obj)调用函数。

这里吊诡的地方在于，para是func的参数，在调用a()的时候，如何能使用para的值呢？

如果理解了作用域链就知道了，这是可以的。首先对于函数func内部的匿名函数，它的作用域链的前端首先是自身的活动对象，下一级是func的变量对象，然后再到全局的变量对象。所以在需要使用para的时候，会一级级的找下去。

这里内部的匿名函数能使用外部函数的变量，这就是闭包的原理。

这里需要注意的是 var a = func(‘test’)，这一句之后，函数对象并没有销毁，因为返回的匿名函数的作用域链还在指向这个活动对象，知道匿名函数销毁后，函数这个活动对象才会被销毁。这也是为什么大量使用闭包，会对性能有影响的原因。

JavaScript 的预编译

JS 每段代码在执行之前，有一个预编译的过程，会首先处理var关键字和function声明。

这里引用鸟哥的描述

在调用函数执行之前, 会首先创建一个活动对象, 然后搜寻这个函数中的局部变量定义,和函数定义, 将变量名和函数名都做为这个活动对象的同名属性, 对于局部变量定义,变量的值会在真正执行的时候才计算, 此时只是简单的赋为undefined.

所以，当我们有：

foo();
//foo
function foo(){
    console.log("foo");
}

这里似乎调用foo()之时，函数都还没有定义，怎么能打印出值呢？这里就是预编译的神奇之处，相当于已经把foo的声明提前了，所以能打印出值。

但是函数表达式却不然，因为函数表达式是一个赋值（计算）的过程，这是要在执行当中才计算的。所以：

foo();
//foo is not defined
foo = function(){
    console.log("foo");
}
