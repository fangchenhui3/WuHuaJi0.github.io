JavaScript几种常见的设计模式

发表于 2015-05-22   |   分类于 前端   |  

工厂模式

function factory(name,age,sex){
var o = new Object();
o.name = name;
o.age = age;
o.sex  = sex;
return o
}

var a = factory("jack",25,"man");
var b = factory("mike",20,"women");

在这里factory函数就是一个工厂，给它传递三个参数，就会返回一个包含有这三个函数的对象，就不用每次手动去生成对象了。
但是工厂模式有一个缺点：不能识别对象的类型。

构造函数模式

function Person(name,age,sex){
this.name = name;
this.age = age;
this.sex = sex;
this.say=function(){
    console.log("hello");
}
}
var p1 = new Person("张三",12,"man");
var p2 = new Person("李四",15,"man");

这里与工厂模式的区别是，直接把age,name ,sex这些付给this对象，那么当我通过 var p1 = new Person() 创建对象的时候，就创建了一个Person类型的对象，并且把参数作为对象的属性赋值给了p1。

那么function Persion(),就作为p1,p2的构造函数，所以p1,p2就有一个constructor属性指向Person

console.log(p1.constructor);
//[Function Person]

与工厂模式相比，就能识别出对象的类型，例如这里就能识别出对象是一个Person类型。

值得注意的是，如果构造函数不是通过new来使用，而是直接调用，那么它与普通函数没有什么区别，this就指向当前环境的对象（如果在对象中，就是某个对象如果在全局中，就指向window）

构造函数的缺点，需要多次创建函数对象。那么可以考虑，能否将每个对象的共同部分抽象出来。那么原型模式就呼之欲出了。

原型模式

要使用原型模式，首先要知道原型是什么意思，具体的可以看我上一篇的总结。这里简单说一下。

当我们创建一个对象时，系统会自动为这个对象创建一个原型对象，并且这个对象的[[prototyype]]属性，指向这个原型对象，与此同时，原型对象里又有[[constructor]]属性指回这个对象。

当对象寻找某个属性是，先在自己的属性里面查找，如果找不到，就向它的原型对象查找。

举个例子：

function Person(name){
this.name = name;
}
Person.prototype.age = 28;
var p1= new Person("mike");
var p2 = new Person("jack");
console.log(p1.age);//28
console.log(p2.age);//28

在这里首先定义了Person函数，然后通过Person.prototype.age = 28 为它的原型对象添加一个属性age，并设置为28。然后在Person的实例p1,p2中访问age属性，实际上并没有，但是它们能通过向上一级，查找原型对象的属性，所以能打印出28。

这里值得注意的是无论是p1,p2还是以后的pn,都共享了age=28这个属性，也就是说如果：

Person.prototype.age=29;

那么p1,p2的age也会变为29。

这与Java之类的语言中的static这个概念很类似，可以把二者联系起来类比。

还需要注意的是，如果单方面修改实例的值，是为实例添加某个属性，而不会影响原型中的值。看代码：

console.log(p1.age);
//28 这里的值来之Person.prototype

Person.prototype.age=29;
//修改原型中的值为29

p1.age=30
//p1并没有age这个属性，这一句实际上是为p1加上age这个属性并赋值30。并不影响原型中的age
console.log(p1.age);
//30
//这里因为p1已经有age这个属性了，不用向原型中查找，所以直接打印30

console.log(p2.age);
//29
//而p2还需要向原型中查找，而原型中的age并改为了29,所以打印29。

介绍到这里，就可以解决了构造对象模式的缺点，把实例之间所共有的属性加入原型对象中，就不用每次创建对象是赋值共有属性了。

其他设计模式
